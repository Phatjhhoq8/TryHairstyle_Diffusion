\section{Triển khai Backend}

\subsection{Cấu hình và khởi tạo}
Backend được xây dựng trên framework \textit{FastAPI}, cung cấp API RESTful hiệu suất cao với khả năng tự động sinh tài liệu Swagger. Cấu hình hệ thống được tập trung tại file \texttt{config.py}, quản lý các đường dẫn model, thư mục dữ liệu và tham số kết nối Redis.

\begin{Verbatim}
# config.py - Cấu hình đường dẫn chính 
BASE_DIR   = Path(__file__).resolve().parent.parent.parent
MODELS_DIR = BACKEND_DIR / "models"
OUTPUT_DIR = BACKEND_DIR / "data" / "output"
UPLOAD_DIR = BACKEND_DIR / "data" / "uploads"
\end{Verbatim}

Biến môi trường được quản lý thông qua file \texttt{.env} với các tham số quan trọng:
\begin{itemize}
    \item \texttt{HUGFACE\_TOKEN}: Token xác thực HuggingFace để tải model gốc.
    \item \texttt{REDIS\_URL}: Địa chỉ kết nối Redis broker (mặc định: \texttt{redis://localhost:6379/0}).
    \item \texttt{API\_HOST} và \texttt{API\_PORT}: Cấu hình host và port cho FastAPI server.
\end{itemize}

\subsection{Triển khai hàng đợi tác vụ}
Celery được sử dụng làm task queue, kết hợp với Redis làm message broker. Worker được cấu hình chạy ở chế độ \texttt{--pool=solo} nhằm đảm bảo toàn bộ pipeline AI được thực thi tuần tự trên một tiến trình duy nhất, tránh xung đột truy cập GPU đồng thời.

Luồng xử lý tác vụ diễn ra như sau:
\begin{enumerate}
    \item FastAPI nhận yêu cầu từ người dùng, lưu ảnh upload vào thư mục session riêng biệt.
    \item Tạo Celery task với các tham số đầu vào, trả về \texttt{task\_id} ngay lập tức cho frontend.
    \item Worker nhận task từ Redis, thực thi pipeline AI với cập nhật trạng thái qua từng giai đoạn (\texttt{PROCESSING}: Face Analysis $\rightarrow$ Creating Mask $\rightarrow$ Estimating Depth $\rightarrow$ Generating Hair).
    \item Kết quả ảnh sinh được lưu vào \texttt{data/output/} và phục vụ qua static file server.
\end{enumerate}

\subsection{Cơ chế cách ly tiến trình GPU}
Để giải phóng triệt để bộ nhớ VRAM sau mỗi lượt suy luận, toàn bộ pipeline Diffusion được khởi chạy trong một tiến trình con (\textit{Subprocess}) cách ly hoàn toàn. Khi tiến trình con kết thúc, hệ điều hành tự động thu hồi toàn bộ tài nguyên GPU mà tiến trình đó đã cấp phát. Cơ chế này đặc biệt quan trọng trên GPU có VRAM hạn chế (12 GB), giúp hệ thống xử lý được nhiều yêu cầu liên tiếp mà không bị tràn bộ nhớ.

\section{Triển khai bằng Docker}

\subsection{Kiến trúc container}
Hệ thống hỗ trợ triển khai thông qua Docker để đơn giản hóa quá trình cài đặt và đảm bảo tính nhất quán môi trường. Kiến trúc Docker bao gồm các container sau:

\begin{itemize}
    \item \textbf{Container Backend (FastAPI + Celery):} Chạy API server và worker AI trên cùng một image, sử dụng NVIDIA Container Toolkit để truy cập GPU từ bên trong container.
    \item \textbf{Container Frontend (Nginx):} Build ứng dụng React thành các file tĩnh, phục vụ thông qua web server Nginx với cấu hình reverse proxy chuyển tiếp các request API đến container backend.
    \item \textbf{Container Redis:} Message broker độc lập, giao tiếp nội bộ với backend qua Docker network.
\end{itemize}

\subsection{Cấu hình Nginx}
Frontend container sử dụng Nginx làm web server, đồng thời đóng vai trò reverse proxy để chuyển tiếp các request có đường dẫn \texttt{/api/*} và \texttt{/output/*} đến backend container. Điều này giúp frontend và backend có thể giao tiếp thông qua một điểm truy cập duy nhất tại cổng 3000, đơn giản hóa việc cấu hình CORS và triển khai.

\subsection{Quy trình triển khai Docker}
Quy trình triển khai hệ thống bằng Docker được thực hiện theo các bước:

\begin{enumerate}
    \item \textbf{Chuẩn bị model:} Do model weights có dung lượng lớn ($\sim$15 GB), cần tải trước trên máy host bằng script \texttt{download\_models.py}. Thư mục \texttt{backend/models/} được mount vào container thay vì copy trực tiếp.
    \item \textbf{Cấu hình biến môi trường:} Sao chép file \texttt{.env.example} thành \texttt{.env}, điền \texttt{HUGFACE\_TOKEN} và chỉnh \texttt{REDIS\_URL} thành \texttt{redis://redis:6379/0} (tên service trong Docker network).
    \item \textbf{Build và khởi động:} Chạy lệnh \texttt{docker compose up --build -d} để build image và khởi động toàn bộ hệ thống.
    \item \textbf{Kiểm tra:} Xác nhận tất cả container đang chạy bằng \texttt{docker compose ps}, kiểm tra log bằng \texttt{docker compose logs -f backend}.
\end{enumerate}

\section{Triển khai chạy trực tiếp (Local)}

Đối với môi trường phát triển, hệ thống có thể được khởi chạy trực tiếp trên máy cục bộ. Quy trình bao gồm:

\begin{enumerate}
    \item \textbf{Thiết lập môi trường:} Tạo virtual environment Python, cài đặt PyTorch với CUDA 12.1 và các dependency từ \texttt{requirements.txt}. Frontend cài đặt qua \texttt{npm install}.
    \item \textbf{Tải model:} Chạy \texttt{python download\_models.py} để tải tự động tất cả model weights từ HuggingFace Hub và Google Drive vào thư mục \texttt{backend/models/}.
    \item \textbf{Khởi động dịch vụ:} Hệ thống cần 4 tiến trình chạy đồng thời trên các terminal riêng biệt:
    \begin{itemize}
        \item \textbf{Redis:} \texttt{redis-server --daemonize yes}
        \item \textbf{FastAPI:} \texttt{uvicorn backend.app.main:app --host 0.0.0.0 --port 8000}
        \item \textbf{Celery Worker:} \texttt{celery -A backend.app.tasks.celery\_app worker --loglevel=info --pool=solo}
        \item \textbf{Frontend:} \texttt{cd frontend \&\& npm run dev}
    \end{itemize}
\end{enumerate}

Ngoài ra, dự án cung cấp script \texttt{start.sh} cho phép khởi động đồng thời Redis, FastAPI và Celery Worker chỉ bằng một lệnh duy nhất: \texttt{wsl -e bash start.sh}. Script này tự động xử lý việc thiết lập biến môi trường, kích hoạt virtual environment và quản lý tiến trình với khả năng dọn dẹp (cleanup) khi nhận tín hiệu dừng.

\section{Triển khai huấn luyện trên Google Colab}

Nhằm đáp ứng yêu cầu tính toán lớn của mô hình khuếch tán SDXL, quá trình huấn luyện hệ thống (Fine-tuning) được triển khai trên nền tảng máy chủ điện toán đám mây Google Colab với cấu hình GPU Tesla T4 (15 GB VRAM). Các tối ưu kỹ thuật chuyên sâu đã được áp dụng để đảm bảo quá trình huấn luyện diễn ra ổn định trong môi trường tài nguyên hạn hẹp:

\subsection{Thiết lập môi trường và cấu hình I/O}
Pipeline huấn luyện được thiết kế dưới định dạng sổ tay Jupyter (\texttt{train\_colab.ipynb}) kết hợp với script xử lý song song (\texttt{train\_stage2.py}). Để khắc phục tình trạng truy xuất dữ liệu chậm khi kết nối trực tiếp với Google Drive (FUSE mount), hệ thống tích hợp cơ chế \textbf{Auto-Caching}:
\begin{itemize}
    \item Sao chép tự động các tập dữ liệu nhỏ (data chunks) từ Google Drive sang ổ đĩa SSD nội bộ (\texttt{/tmp/}) của Colab trước mỗi vòng lặp.
    \item Hạn chế thao tác I/O trong suốt quá trình đồng bộ (forward/backward pass), giúp tăng tốc độ nạp dữ liệu (data loading) lên gấp 5--10 lần.
\end{itemize}

\subsection{Quản lý bộ nhớ (VRAM Management)}
Với dung lượng 15 GB VRAM trên Tesla T4, việc huấn luyện mô hình SDXL với cấu trúc UNet 13-kênh dễ gây ra hiện tượng tràn bộ nhớ (Out-Of-Memory). Để khắc phục, hệ thống áp dụng các kỹ thuật cấu hình bộ nhớ cấp thấp:
\begin{enumerate}
    \item \textbf{Tái cấu trúc đồ thị tính toán (Gradient Checkpointing và Expansion):} 
    Hệ thống áp dụng kỹ thuật Gradient Checkpointing nhằm giảm thiểu lượng bộ nhớ cần thiết trong quá trình lan truyền ngược (backpropagation). Đồng thời, cấu hình biến môi trường \texttt{PYTORCH\_CUDA\_ALLOC\_CONF="expandable\_segments:True"} được sử dụng để cải thiện cơ chế cấp phát bộ nhớ của PyTorch, giúp hạn chế hiện tượng phân mảnh VRAM trong quá trình huấn luyện.

    \item \textbf{Cơ chế trút tải bộ nhớ (Offloading):} 
    Các thành phần không yêu cầu tính toán gradient, chẳng hạn như bộ mã hóa văn bản (Text Encoder) hoặc VAE đã huấn luyện trước, được chuyển xuống bộ nhớ RAM của CPU. Khi cần thiết, các thành phần này sẽ được nạp lại vào GPU theo từng bước, qua đó giảm đáng kể mức sử dụng VRAM tại mỗi thời điểm.

    \item \textbf{Huấn luyện tham số hiệu quả (LoRA 8-bit):} 
    Toàn bộ mạng UNet xương sống được cố định trọng số, chỉ cho phép cập nhật các lớp mở rộng đầu vào (như \texttt{conv\_in}) và các khối Cross-Attention Injector. Kết hợp với bộ tối ưu hóa AdamW ở chế độ 8-bit, phương pháp này giúp giảm số lượng tham số cần huấn luyện xuống dưới 1\%, từ đó tiết kiệm bộ nhớ và tăng tốc độ huấn luyện.
\end{enumerate}

\subsection{Cơ chế sao lưu linh hoạt (Checkpoint Fallback)}
Do Google Colab thường xuyên ngắt kết nối (session timeout) đối với các tác vụ dài, luồng huấn luyện được tích hợp hệ thống bắt tín hiệu (Signal Handling) dựa trên \texttt{SIGINT} và \texttt{SIGTERM}. Khi phát hiện tín hiệu gián đoạn, mô hình tự động đóng gói trạng thái hội tụ (weights, optimizer state, step counter) vào bộ đệm \texttt{/tmp/}. Để bảo vệ dữ liệu vĩnh viễn, toàn bộ checkpoint được hệ thống tải tự động lên một kho lưu trữ an toàn (HuggingFace Hub) sau mỗi Epoch, cho phép tiếp tục huấn luyện kể cả khi phiên Colab đổi tài khoản hoặc reset hoàn toàn.
